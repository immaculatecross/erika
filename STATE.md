**v0.2 IN PROGRESS — the coaching layer over v0.1's findings (D-15), dispatched E-7 → E-9 → E-11 → E-6 → E-12.**

- **E-11 Speech archive — DONE.** An `/archive` screen (a new 6th sidebar item, before Settings) is your speaking life at a glance: every analyzed moment in chronological order, newest session first, grouped by session, each row showing what you said (the `quote`), its category, severity and a timestamp — with a jump-to-audio that opens the source session at that moment. Model-light — no model calls, no migration, no writes; a time-organized view over existing findings (it complements E-9's Phrasebook, which organizes the same findings as a recast reference). `lib/archive.ts` is the pure, client-safe core (unit-tested): `buildEntries` builds the timeline (session `createdAt` newest-first, then `startMs` ascending within a session — spoken order, consistent with the analysis report; ties by sessionId then id, deterministic), `filterEntries` intersects a free-text query (case-insensitive over quote/correction/explanation) with a category AND a severity (blank query → all; "all" → no constraint), and `groupBySession` collects the ordered entries into per-session groups. The new `listAllFindingsWithSession` (in `findings.ts`) joins each finding to its session's `created_at`/`original_filename`; `GET /api/archive` serves the built timeline. Jump-to-audio is a `/sessions/[id]?t=<startMs>` deep link — the session detail page reads `?t=` on mount and seeks the reused player once audio metadata is ready, landing exactly like the in-page report's `onJump`. DESIGN — calm chronological rows and session/day headers, ink accent, green/red only as meaning (severity badges), tabular timestamps; a plain search input and quiet category/severity chips; one signature stagger on entry; quiet empty state until a session is analyzed.
- **E-7 Focus map & progress metrics — DONE.** A `/focus` screen answers on one screen: how often do I make each kind of mistake, is it getting better, and what should I work on next. Pure aggregation, model-light — no model calls, no writes, no migration; it reads existing findings, speech segments and analysis jobs. `lib/focus.ts` is the pure metric math (unit-tested against hand-computed fixtures): **speech-hours** = Σ segment `durationMs` over *analyzed* sessions (analysis `done`) ÷ 3.6M; **category error rate** = findings ÷ speech-hours per category (zero-filled, never NaN); **trend** = the rate bucketed per analyzed session chronologically, direction = earliest vs most-recent bucket (falling = improving); **"what to work on next"** ranks by severity-weighted rate (Σ(weight×count)÷hours, high 3 · medium 2 · low 1), tie broken by category order. `GET /api/focus` serves the model; the screen renders a hero error-rate, a hand-rolled monochrome **SVG** sparkline (`components/sparkline.tsx`) + trend badge, and a ranked bar list (`components/category-bars.tsx`) — green/red only where trend carries meaning (D-14), tabular numerals, quiet empty state when nothing is analyzed. The formula is documented on-screen.
- **E-9 Phrasebook / the recast library — DONE.** A `/phrasebook` screen (a new sidebar item, before Settings) is a searchable library of every recast built from findings — `quote` (you say) beside `correction` (natives say), plus the why, category and severity in calm two-column rows. Model-light — no model calls, no migration, no analysis/ingest changes. `lib/phrasebook.ts` is the pure, client-safe core (unit-tested): `buildEntries(findings, inDeckSet)` and `filterEntries` — a free-text query (matched case-insensitively against quote/correction/explanation) AND a category, intersected; a blank query returns all. `listAllFindings` (new, in `findings.ts`) reads the whole library across sessions; `GET /api/phrasebook` serves entries each marked in-deck (a card exists for the finding). **Pin** (`POST /api/phrasebook/[findingId]/pin`) calls the new **additive** `createCardForFinding(db, findingId)` — the single-finding seam that **clears any `deleted_findings` tombstone** then idempotently inserts the card (INSERT OR IGNORE on the `finding_id` UNIQUE key) — so a card the user deleted (E-5b) can be deliberately added back and returns to `listDueCards`. The bulk `generateCards`, the SM-2 scheduler, and the grade/due flow are untouched. Green/red only as meaning (severity badges, the subtle "in deck" check); plain search input + quiet category chips.
