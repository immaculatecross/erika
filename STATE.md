**E-2 Capture shipped.** Both ways in are live. Upload backbone (part 1): `POST /api/sessions` streams an audio file to disk under `data/sessions/<id>/` (never buffered — a byte cap aborts mid-stream), `ffprobe` probes duration (D-7; over-24 h/2 GB rejected and cleaned up), and it becomes a session row with a `queued` ingest job. Mic capture (part 2): a Record control beside Upload records live from the mic — a chunked `MediaRecorder` (1 s timeslices, so long takes survive), an elapsed timer in tabular numerals, and the signature level meter that breathes with the voice off a Web Audio `AnalyserNode` (spring on `scaleY`, degrading to a non-animated height readout under `prefers-reduced-motion`). On Stop the ordered chunks assemble, are decoded and re-encoded to WAV client-side (a live WebM carries no container duration ffprobe can read), and POST to the same endpoint — landing in the list with a `queued` job. Denied/unsupported mic renders a quiet, specific line; Upload keeps working. Pure logic (elapsed format, level-from-analyser, chunk assembly, mime pick, WAV encode) lives in `lib/recording.ts` (unit-tested); the browser wiring is `lib/use-recorder.ts` + `components/recorder.tsx` + `components/level-meter.tsx`; both capture paths share `lib/upload-audio.ts`. The Sessions screen lists takes (newest-first, tabular duration); `/sessions/<id>` plays via `GET …/audio` with HTTP Range (206) and deletes rows + files. Migration v2 adds `sessions` + `ingest_jobs` (FK cascade). Gates `lint`/`typecheck`/`test`/`build`/tripwires are green (CI installs ffmpeg); a fake-audio-device Playwright suite drives Record → Stop → session end to end.

**E-3 Smart ingest shipped (both parts).** Captures are processed into deduplicated, timestamped speech segments with triage renditions — resumably and memory-safely — and the session page surfaces the result. Part 1, the pipeline: `npm run worker` drains `queued` jobs through a testable `processJob(db, jobId)` — normalize to 16 kHz mono, energy-based VAD via ffmpeg `silencedetect` (parse → invert → merge ~300 ms gaps → drop sub-2 s), persist each segment with original-timeline timestamps + a streamed SHA-256 content hash (migration v3 `segments`), and produce a configurable (`TRIAGE_TEMPO`, 1.25–1.5×, default 1.5) time-compressed rendition cached by hash under shared `data/cache/`. Every audio op is ffmpeg file→file (memory-safe on day-scale input); jobs are checkpointed via `ingest_jobs.stage`/`progress`/`error` so a killed worker reclaims its `processing` job and resumes without redoing stages or duplicating segments; failures land `failed` with a truthful message. Part 2, the read-only UI: a `GET /api/sessions/[id]/ingest` route returns `{ state, stage, progress, error }` plus a raw-vs-speech summary and the segments; the detail page polls it while queued/processing (a restrained transform-only progress bar, not E-4's orb), then transitions to the result without a manual reload and stops polling on a terminal state. Done shows raw-vs-speech time ("6h 2m → 47m speech", tabular, `lib/ingest-view.ts` pure helper) and a monochrome segment timeline whose blocks sit at proportional offsets and seek the reused audio player on click; a `failed` job shows its stored error, a `done` job with no segments says so quietly. Additive read-only work — no pipeline/worker/migration change. Verified end to end (real worker run for part 1; 74 unit tests + a Playwright suite that drives a job `processing → done` and asserts the live no-reload update, that polling stops, proportional offsets, seek, and the failed/empty states).
